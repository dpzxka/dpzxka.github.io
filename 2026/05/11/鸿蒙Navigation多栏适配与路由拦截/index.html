鸿蒙 Navigation 多栏适配与路由拦截：从折叠屏到平板的全场景方案

|总字数:5.6k|阅读时长:24分钟|浏览量:

在前面的三篇文章中，我们完整覆盖了 NavDestination 组件的核心知识——基础路由操作、HomePathInfo 主页集成方案、生命周期深度解析。但如果你的应用只跑在手机上，那这些内容其实已经够用了。

然而鸿蒙生态的特殊之处就在于——你的应用可能运行在手机、折叠屏、平板甚至车机上。同样的代码，在 360vp 的手机上和在 1000vp+ 的平板上，布局体验是天差地别的。

这篇文章就来解决这个问题。我们会聊两件事：

  1. Navigation 的多栏自适应模式——如何用一套代码同时适配 Stack（单栏）和 Split（双栏）布局
  2. 路由拦截机制——如何在页面跳转前后插入权限校验、埋点统计等逻辑

这两个能力加在一起，就构成了 Navigation 框架在复杂设备形态下的完整控制力。

顺便说一下，本文所有代码示例均采用 V2 装饰器（@ComponentV2 / @Local / @Param / @Monitor），这是 API 12+ 推荐的组件开发模式。

一、Navigation 的三种模式

Navigation 组件提供了三种模式来控制页面区域的展示方式：

Stack 模式：导航栏和内容区独立，每次页面切换都像手机 App 一样——新页面推入栈顶，覆盖上一页。

Split 模式：左侧显示导航栏（NavBar），右侧显示内容区。点击左侧菜单项，右侧切换对应的 NavDestination。两边独立显示，可以同时看到。

Auto 模式：框架根据 @Local windowWidth 自动判断——≥600vp 用 Split，<600vp 用 Stack。

注意，这个 600vp 的阈值是系统内置的，不需要我们配置。但你可以在运行时通过 setInterception 的 modeChange 回调监听模式切换。

但这里有一个容易被忽视的点：在 Split 模式下，左侧 NavBar 里显示什么？

答案是：NavBar 的内容就是 NavDestination 栈中栈底页面的导航栏标题和返回按钮。也就是说，如果你 push 了 HomePage、CategoryPage、CartPage、ProfilePage 等多个页面到栈中，左侧 NavBar 会显示当前选中项的标题和返回按钮。

如果你想要更精细地控制 NavBar 里展示的内容（比如自定义菜单项列表），可以通过 Navigation 的 navBarWidth、hideNavBar、自定义 Navigation 标题栏等属性来实现。

2.3 折叠屏的特殊处理

折叠屏有一个手机和平板都没有的中间状态——半折叠。屏幕一半在桌面、一半翘起。在折叠屏上，我们不能简单地用 600vp 一刀切。

当设备折叠时，屏幕宽度可能只有 400vp（外屏或折叠态），应该走 Stack 模式；展开后宽度可能达到 800vp+，应该走 Split 模式；而在半折叠态，屏幕宽度的计算方式会有所不同。

三、路由拦截：Navigation 的”门禁系统”

聊完多栏适配，我们进入第二个话题——路由拦截。

很多时候，我们不希望用户随意跳转页面：未登录用户不能进入支付页、管理员才能访问后台、从某个特定页面返回时需要二次确认……这些统称为路由守卫。

Navigation 通过 NavPathStack.setInterception() 方法提供了三个维度的拦截能力：

回调 触发时机 典型用途
willShow 页面跳转前触发，允许在此时操作栈（push/pop/replace） 权限校验、登录拦截、参数预处理
didShow 页面跳转完成后触发，此时栈操作不再影响当前跳转 埋点统计、日志记录、页面访问追踪
modeChange 单双栏切换时触发 布局适配、UI 状态同步

六、总结

在这篇文章中，我们深入了两个 Navigation 的高级话题：

多栏适配解决的是”一套代码跑在所有设备上”的问题。核心思路是通过 NavigationMode.Auto 或手动监听设备状态来切换 Stack/Split 模式，再配合 navBarWidth、自定义 NavBar 内容等属性来精细控制布局。

路由拦截解决的是”页面跳转的控制权”问题。setInterception 的三个回调覆盖了跳转前（willShow）、跳转后（didShow）、布局变化（modeChange）三个关键时机，非常适合做登录校验、权限控制、埋点统计和布局同步。

这两者结合在一起，能让 Navigation 应用在复杂的设备环境中保持稳定和可控。
