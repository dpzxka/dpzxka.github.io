鸿蒙 Navigation 完全解析，带你从源码的角度彻底理解路由导航

发表于2026-05-09|更新于2026-05-09|harmonyos
|字数总计:4.7k|阅读时长:16分钟|阅读量:

其实我一直准备写一篇关于鸿蒙 Navigation 的文章。在我经过较长时间的筹备之后，终于决定开始动手了。

目前虽然网上相关的文章也不少，但我觉得没有哪篇写得特别详细的。大多数文章要么只讲了基础用法，要么笼统地过了一遍 API，很少有人把 Navigation 从入门到原理完整串起来的。所以我决定自己写一篇，希望能帮到正在学习鸿蒙开发的读者。

Navigation 是鸿蒙应用开发中最重要的导航组件之一。如果你是从 Page 跳转的 router.pushUrl 时代过来的，你会发现 Navigation + NavPathStack 这套体系无论是页面管理能力、生命周期控制还是多端适配支持，都远远强于传统的 router 方案。

阅读本文讲究由浅入深，循序渐进。我们先从简单的开始，逐步深入到源码层面。

一、Navigation 是什么？先看整体架构

Navigation 是鸿蒙提供的路由导航根视图容器。一般作为 Page 页面的根容器使用，内部默认包含了标题栏（Title Bar）、内容区（Content Area）和工具栏（Toolbar）三个区域。

下图展示了 Navigation 的整体架构：

从图中可以看到，Navigation 内部由三大区域组成：

  • 标题层：支持主标题/副标题配置，提供 Mini、Free、Full 三种显示模式
  • 内容层：动态路由区，通过 NavDestination 承载页面内容。首页可以放 Navigation 的子组件，非首页放 NavDestination 的子组件
  • 交互层：集成工具栏，提供搜索、收藏、分享等标准化操作入口

Navigation 的路由推荐基于 NavPathStack 来实现。从 API Version 10 开始，官方推荐使用 NavPathStack 配合 navDestination 属性来进行页面路由管理。

四、NavDestination 生命周期深入分析

Navigation 相比传统 router 方案，一个很大的优势在于生命周期管理的精细度。NavDestination 拥有一套完整的生命周期回调，让你可以精确控制页面在不同阶段的行为。

4.1 生命周期全览

NavDestination 提供了以下几个生命周期回调：

回调 触发时机 典型用途
aboutToAppear 组件构建前执行 初始化数据、修改状态变量
onWillAppear （API 11+） 组件挂载前执行 状态变更立即生效，可用于最终确认
onShown 布局完成后触发 启动动画、数据加载完成后的渲染
onHidden 页面被覆盖或进入后台 暂停视频、停止定时器
onWillDisappear 销毁前执行 资源释放、取消订阅
onNewParam （API 19+） Single 模式收到新参数 刷新页面数据

4.2 页面栈切换时的生命周期流转

为了更好地理解，我们来看一个实际的页面切换场景。

五、路由拦截与高级特性

Navigation 还提供了一些高级功能，让你的路由控制更加灵活。

5.1 路由拦截（setInterception）

这部分是很多开发者容易忽略的，但其实特别实用。你可以通过 setInterception 拦截每一次路由操作，在跳转发生前进行权限校验、数据保护等操作。

这个方法返回一个拦截结果对象，支持以下字段：

  • proceed: boolean — 是否放行该路由操作
  • redirect: string — 重定向到其他页面
  • params: object — 重定向时携带的参数

拦截机制的应用场景很多。比如用户没有登录时访问个人中心 → 自动重定向到登录页；比如编辑页面有未保存内容 → 弹窗提示是否保存后再离开。

六、三种导航模式与多端适配

鸿蒙系统的特色之一就是支持多种设备形态。Navigation 天然支持三种导航模式，让同一套代码可以无缝跑在手机、平板和折叠屏上。

6.1 Stack 模式（单栏）

适用设备：手机、手表

Stack 模式下，每次 pushPath 都会全屏显示新的页面，当前页面被覆盖到下层。用户按返回时，一层一层弹出。这也是手机上最熟悉的导航方式。

6.2 Split 模式（分栏）

适用设备：平板、折叠屏展开态

Split 模式下，页面分为左右两栏，左侧是导航栏（通常用于列表），右侧是内容区。点击左侧列表项，右侧展示对应的详情内容。

这种模式在邮件应用、设置页面、文档浏览器等场景中非常常见。

6.3 Auto 模式（自适应）

适用设备：全设备

Auto 模式是追求多端适配的首选方案。系统会根据屏幕宽度自动切换布局：

  • 屏幕宽度 < 600vp → 自动采用 Stack 模式
  • 屏幕宽度 ≥ 600vp → 自动采用 Split 模式

这个阈值是鸿蒙系统自动判断的，你不需要写额外的适配代码。手机端打开时是 Stack，平板或折叠屏展开时自动变成 Split，页面组件完全复用。

6.4 多端适配的最佳实践

对于追求多端体验的应用，建议的做法是：

  1. 所有页面都写成 NavDestination 形式
  2. Navigation 的 mode 设置为 Auto
  3. 借助 NavigationMode 的当前值，在页面内部做微调

这样做的好处是：同一套页面组件，手机端 Stack 全屏展示，平板端 Split 分栏展示，后续如果要适配折叠屏、PC 等新形态，也不需要大规模重写页面。

七、源码分析：Navigation 组件的工作原理

好了，基础用法和高级特性我们都讲完了。接下来我们深入到源码层面，看看 Navigation 内部的运作机制。这部分我会尽量用通俗的语言来解释。

7.1 Navigation 的渲染流程

Navigation 的本质是一个 容器组件，它内部维护了一个页面栈结构。当你调用 pushPathByName 时，整个流程是这样的：

  1. 查找路由映射：根据传入的 pageName，从 route_map.json 或 @Builder 映射表中查找对应的页面组件
  2. 入栈操作：将页面信息（name, param, 构建函数）封装为一个栈节点，push 到 NavPathStack 内部数组
  3. 刷新渲染：触发 Navigation 组件的重新渲染，新的 NavDestination 被构建出来
  4. 生命周期触发：新页面走 aboutToAppear → onWillAppear → onReady → 构建 → onShown
  5. 旧页面隐藏：如果栈中原有页面被覆盖，触发其 onHidden 回调

7.2 NavPathStack 的内部结构

NavPathStack 内部实际上维护了一个 Array<NavPathInfo> 类型的数组。每个 NavPathInfo 包含了页面的名称、参数、构建函数等元信息。

7.4 Navigation vs Router 的选择

听到这里，可能有人会问：既然 Navigation 这么强大，那老的 router 方案（router.pushUrl）是不是该彻底弃用了？

我的看法是这样的：

推荐使用 Navigation 的场景：

  • 应用有 3 个以上页面
  • 需要精细的生命周期管理
  • 未来可能要适配平板/折叠屏
  • 需要路由拦截、DIALOG 等高级功能

仍然可以使用 router 的场景：

  • 非常简单的 1~2 个页面的工具类应用
  • 与已有旧代码保持一致
  • 页面之间没有复杂的参数传递和状态共享需求

不过从鸿蒙的发展趋势来看，Navigation 已经是官方主推的路由方案了。新的 API（比如 API 20 新增的 HomePathInfo）还在不断强化 Navigation 的能力。建议新项目直接使用 Navigation。

八、总结

至此，鸿蒙 Navigation 的主要内容已经全部讲解完了。我们来快速回顾一下本文的核心知识点：

  1. Navigation 是鸿蒙的路由导航根容器，包含标题层、内容层、交互层三大区域
  2. NavPathStack 是路由操作的核心类，统一管理 pushPath、pop、replacePath 等操作
  3. NavDestination 有完整的生命周期，aboutToAppear → onWillAppear → onShown → onHidden → onWillDisappear
  4. setInterception 可以实现路由拦截，用于权限校验、数据保护等场景
  5. Three 三种导航模式（Stack/Split/Auto）让同一套代码适配多端
  6. 路由映射表和跨模块路由为中大型应用提供了架构基础

Navigation 这套方案设计得相当成熟。如果你是从 Android 开发转过来的，你会发现它的 NavPathStack 设计思想和 FragmentManager 有些神似；如果你是从 iOS 过来的，它的 UINavigationController 的影子也能在这里找到。但在此基础上，鸿蒙加入了多端自适应、DIALOG 模式等特色能力，让路由导航这件事变得更灵活了。
